# 三目並べ（Tic-Tac-Toe）

複数の人が参加でき、2人ペアが成立した時点で順次対戦できる三目並べです。
HTTP サーバ・コントローラ・GUI クライアントの3層構成になっています。
すべて Python 標準ライブラリのみで動作し、追加インストールは不要です。

## ファイル構成

| ファイル | 役割 |
| --- | --- |
| `controller.py` | ゲームのルール・状態管理・部屋（ペア）の振り分け（HTTP に非依存） |
| `server.py` | HTTP サーバ。リクエスト受付・認証・JSON 変換 |
| `client.py` | tkinter による GUI クライアント |

## 必要環境

- Python 3.7 以上
- `tkinter`（通常は Python に同梱。Linux で未導入の場合は別途インストール）
  - Debian / Ubuntu: `sudo apt install python3-tk`

## 使い方

### 1. サーバを起動する

環境変数で次の設定ができます。

- `ACCESS_TOKEN` … アクセストークン。設定するとリクエストにトークン一致が必須。
- `PORT` … 待ち受けポート番号（未設定なら `8000`）。
- `HOST` … 待ち受けアドレス（未設定なら `0.0.0.0`。全インターフェースで待受）。

Linux / macOS:

```bash
ACCESS_TOKEN=secret123 PORT=9000 python server.py
```

Windows (PowerShell):

```powershell
$env:ACCESS_TOKEN="secret123"; $env:PORT="9000"; python server.py
```

トークンを設定せずに起動すると、認証なし（誰でも接続可）モードで動作します。

```bash
python server.py
```

起動すると待ち受けアドレスと認証モードが表示されます。

```
Server running at http://0.0.0.0:8000  [認証あり]
```

#### ランダムなアクセストークンの生成

暗号学的に安全なトークンは `secrets` モジュールで生成できます。

```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

生成した値を `ACCESS_TOKEN` に設定し、同じ値を各クライアントに入力します。

### 2. クライアントを起動する

対戦したい人数ぶん起動します（同一マシンでも別マシンでも可）。

```bash
python client.py
```

### 3. 接続して対戦する

1. 設定画面で **接続先 URL**（既定 `http://127.0.0.1:8000`）を入力します。
2. サーバでトークンを設定した場合は **Access Token** に同じ値を入力します。
3. **「接続して参加」** を押すと、自動的に空きのある部屋へ割り当てられ、
   マークが付与されます（部屋内の先着が `X`、次が `O`）。
4. 2人揃うまでは盤面が無効化され「対戦相手の参加を待っています...」と表示されます。
5. 同じ部屋に2人揃うと自動的に対局が始まります。自分の手番のときだけ
   マスをクリックできます。
6. 勝敗が決まると結果が表示され、盤面が無効化されます。

### 複数ペアでの対戦

参加するたびに、空きのある部屋（1人しかいない部屋）があればそこへ、
なければ新しい部屋が作られます。来た順に2人ずつ自動でペアになるため、
何人接続していても、2人揃った部屋から順に独立して対戦が進みます。

例: 4人が順に参加した場合
- 1人目 → 部屋1 の `X`（待機）
- 2人目 → 部屋1 の `O`（部屋1 開始）
- 3人目 → 部屋2 の `X`（待機）
- 4人目 → 部屋2 の `O`（部屋2 開始）

## 各プログラムの説明

### controller.py

ゲームの中核ロジックを担います。HTTP には一切依存しないため、
単体テストが容易です。2つのクラスで構成されます。

- `Game` … 1つの対戦（2人ぶん）の盤面・手番・勝者を管理します。
  - 着手処理では、2人揃う前・手番違い・ゲーム終了後・使用済みマスなどの
    不正な手をすべて拒否します。
  - 勝敗判定は縦・横・斜めの8ラインをチェックし、全マス埋まりで引き分けです。
- `Controller` … 複数の `Game`（部屋）を管理し、参加者を2人ずつ振り分けます。
  - `join()` … 空き部屋を探して（なければ新規作成して）プレイヤーを追加し、
    ランダムなトークン・所属部屋・マークを返します。
  - `move(token, pos)` … トークンから所属部屋を特定して着手します。
  - `state(token)` … トークン指定でその部屋の状態、無指定で全体サマリを返します。
  - 状態スナップショットには `game_id` / `board` / `turn` / `winner` /
    `ready`（2人揃ったか）/ `players`（その部屋の人数）が含まれます。

### server.py

`http.server` をベースにした HTTP サーバです。
`ThreadingHTTPServer` で複数クライアントを同時に処理し、状態更新は
`threading.Lock` で直列化して競合を防ぎます。

提供するエンドポイント:

| メソッド | パス | 説明 |
| --- | --- | --- |
| POST | `/join` | ゲームに参加し、トークン・部屋番号・マークを取得 |
| POST | `/move` | 着手する（JSON: `{"player": token, "pos": 0-8}`） |
| GET | `/state?player=<token>` | 自分の部屋の状態を取得（無指定で全体サマリ） |
| POST | `/reset` | 全部屋を初期化 |

認証:

- 環境変数 `ACCESS_TOKEN` が設定されている場合のみ認証を要求します。
- クライアントは `Authorization: Bearer <token>` または `X-Access-Token`
  ヘッダでトークンを送ります。
- トークンの比較は `hmac.compare_digest` による定数時間比較で行い、
  タイミング攻撃を防ぎます。
- 不一致の場合は `401 unauthorized` を返してアクセスを拒否します。

ネットワーク設定:

- `HOST`（既定 `0.0.0.0`）と `PORT`（既定 `8000`）を環境変数で指定できます。
- Render などの PaaS では、プラットフォームが注入する `PORT` を使い、
  `0.0.0.0` でリッスンする必要があります（`PORT` を固定指定しないこと）。

### client.py

tkinter による GUI クライアントです。通信処理（`ApiClient`）と
画面処理（`TicTacToeGUI`）に分かれています。

- **設定画面**: 接続先 URL と Access Token を入力します。
- **ゲーム画面**: 3×3 のボタンで盤面を表示し、状態と部屋番号をステータス欄に
  表示します。
- 状態は1秒ごとに `/state?player=<自分のトークン>` でポーリングし、
  自分の部屋の盤面だけを自動更新します。他の部屋の対戦には影響されません。
- ネットワーク処理は別スレッドで実行するため、通信中も UI が固まりません。
- 2人揃うまでは全マスを無効化し、待機メッセージを表示します。
- 対局中は自分の手番かつ空いているマスのみクリック可能です。

## 通信フローの概要

```
[クライアント] --join--> [server] --> [controller] (空き部屋へ割当、トークン発行)
[クライアント] --state(player)--> [server] --> [controller] (自分の部屋を取得、毎秒)
[クライアント] --move---> [server] --> [controller] (トークンから部屋特定・検証)
```

## 補足・拡張のヒント

- プレイヤートークンは参加ごとに `secrets.token_urlsafe` でランダム発行されます。
- `0.0.0.0` で公開する場合は誰でも URL に到達できるため、`ACCESS_TOKEN` を
  必ず設定することを推奨します。
- さらに拡張するなら、ロングポーリングや WebSocket による即時更新、
  HTTPS 化、対戦終了後の自動マッチング再開、明示的なルーム指定参加などを
  追加できます。
